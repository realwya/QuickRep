# 纯文本健身笔记 iOS App 产品方案 v1

## 1. 产品概述

### 1.1 产品定位
这是一款面向规律训练者的 iPhone 健身记录 App。

它的核心特点不是表格式录入，而是让用户像写笔记一样记录训练内容，系统再从文本中自动提取结构化训练数据。第一版优先服务力量训练 / 增肌训练场景，强调训练中可快速记录，也支持先生成一个轻量训练模板再执行。

### 1.2 核心价值
第一版重点解决两件事：

- 训练时记录足够快，不打断节奏
- 训练后回看某个动作的历史数据足够高效

产品差异化来自三点：

- 文本输入的自然感
- 少量明确语法带来的可靠结构化提取
- 累进式完成进度所带来的执行感

### 1.3 第一版不做什么
以下能力明确不纳入 v1：

- 高级分析图表
- PR / 1RM / 训练量趋势等复杂统计
- 复杂周期计划系统
- 过于宽松的自然语言识别
- Apple Watch / HealthKit 集成

## 2. 核心交互设计

### 2.1 训练记录模型
- 每次训练都是一篇新的笔记
- 用户可以从空白笔记开始，也可以从模板创建训练笔记
- 笔记正文承载原始内容和最终训练结果；训练中的完成进度以独立草稿状态保存，直到用户结束训练

### 2.2 动作输入方式
- 通过 `@动作名` 输入动作
- 输入时提供自动补全
- 从某个动作开始，到下一个动作出现之前的所有内容，都归属于当前动作

示例：

```text
@卧推
20 x 8 x 5
今天状态一般，最后两组比较吃力

@上斜哑铃卧推
24 x 10 x 4
```

上面这段内容中：
- `卧推` 和 `上斜哑铃卧推` 会被识别为两个动作
- `20 x 8 x 5` 和 `24 x 10 x 4` 会被识别为结构化训练数据
- 其他文字会作为备注保留在对应动作下

### 2.3 数据输入语法
第一版采用半结构化输入策略，只支持少量明确语法。

核心语法：

```text
20 x 8 x 5
```

解释为：

- 重量：20
- 次数：8
- 组数：5

系统会将这一行识别为一条计划行，并从中提取目标重量、次数和组数，用于驱动右侧完成进度。

### 2.4 组完成机制
- 每条可识别的计划行右侧，都会出现一个独立的圆形完成按钮
- 例如输入 `20 x 8 x 5` 后，该行右侧会出现一个贴近屏幕边缘的圆形按钮
- 每点击一次表示完成 1 组，按钮会逐步显示圆形进度
- 进度上限严格等于该行的目标组数；`20 x 8 x 5` 最多点击 5 次
- 当点击次数达到目标组数时，进度达到 100%，表示这条计划行的预期组数已全部完成
- 同一动作下如果存在多条计划行，每一行各自拥有独立的完成按钮和进度
- 每次点击后，系统只更新该行右侧圆圈的草稿进度，不改写正文

例如用户先输入：

```text
20 x 8 x 5
```

系统会在这一行右侧显示一个圆形按钮。用户每点击一次，表示完成了 1 组；圆圈数字和环形进度会递增，但正文仍保持 `20 x 8 x 5` 不变。点击 5 次后，该行完成进度达到 100%。

进度与编辑规则：

- 进度只绑定该计划行本身，不绑定该行下方的自由备注
- 修改备注内容不会影响该计划行的完成进度
- 训练中的完成进度以独立草稿状态为事实源，而不是正文后缀
- 如果修改计划行内容本身，原完成进度直接清空，并按新的计划重新记录
- 计划行在编辑过程中如果暂时变成非法格式，按钮先保留不变
- 只有当用户离开这一行后，系统才重新判断它是否仍然是合法计划行
- 如果离开该行后格式合法，则保留按钮并按新内容重建进度规则
- 如果离开该行后格式不合法，则移除按钮并清空该行草稿进度

结束训练时的收敛规则：

- `20 x 8 x 5` 且实际完成 4 组时，最终保存为 `20 x 8 x 4`
- `20 x 8 x 5` 且实际完成 5 组时，最终保存为 `20 x 8 x 5`
- `20 x 8 x 5` 且未完成任何一组时，结束训练时直接删除
- 最终历史记录只保留实际完成结果，不保留原目标组数痕迹

### 2.5 自由文本处理
- 未命中结构化语法的文字全部保留
- 这些内容不会被强行解析
- 文本会作为笔记备注挂在当前动作下，或作为整篇训练的上下文内容存在

这点非常关键，因为它保证产品仍然是“笔记优先”，而不是被迫变成表单输入工具。

## 3. 模板设计

### 3.1 模板定位
第一版支持轻量训练模板，但模板本质上仍然是一篇文本笔记，而不是独立的复杂计划系统。

### 3.2 模板能力范围
第一版模板支持：

- 预填训练动作
- 预填目标组数 / 次数 / 重量结构
- 训练时按计划行累计记录完成进度，并在圆圈中实时反馈
- 训练中临时调整动作或计划行内容

不支持：

- 周期计划
- 多周推进逻辑
- 自动递增规则
- 复杂训练编排

### 3.3 模板示例
```text
@深蹲
100 x 5 x 5

@罗马尼亚硬拉
80 x 8 x 4

@腿举
140 x 12 x 4
```

用户开始训练后，这篇模板会直接变成当天的训练笔记，训练时在此基础上执行、记录完成进度并修改内容。

## 4. 页面与信息架构

### 4.1 首页
首页以“开始训练”和“回看最近训练”为核心。

建议包含：

- 开始新训练
- 从模板开始
- 最近训练列表

首页不承担复杂数据仪表盘职责，避免第一版过重。

### 4.2 训练编辑页
训练编辑页是产品核心页面，负责承载主要体验。

核心能力：

- 文本输入
- `@动作` 自动补全
- 结构化数据识别
- 计划行识别
- 每条计划行的独立完成按钮与圆形进度
- 基于计划行内容的进度重置与重建
- 完成进度作为独立草稿状态实时反馈
- 结束训练时按实际完成情况收敛计划行
- 保留自由备注

设计原则：

- 用户尽量少跳页
- 输入、查看、完成动作尽量在同一页面完成
- 优先保证边练边记时的速度和清晰度

### 4.3 动作详情页
每个动作都有一页独立详情页。

第一版重点展示：

- 该动作的历史记录列表
- 每次训练的日期
- 实际完成的计划行记录与完成情况
- 当次备注

第一版不把详情页做成复杂分析页，而是优先满足快速复盘。

### 4.4 动作库页
动作库用于支撑输入体验和长期管理。

第一版包含：

- 搜索动作
- 新增动作
- 编辑动作
- 删除动作
- 支持默认内置常用动作
- 支持用户自定义动作

动作库的目标不是做得极其庞大，而是保证 `@动作` 自动补全体验足够顺滑。

### 4.5 设置页
第一版设置页保持极简。

仅保留必要配置，例如：

- 全局重量单位：`kg` / `lb`

第一版不加入过多偏好设置，避免增加理解成本。

## 5. 数据与对象模型

### 5.1 核心对象
第一版核心对象建议只有以下几类：

- 训练笔记
- 动作
- 计划行
- 模板
- 动作库条目

### 5.2 数据归属逻辑
- 一篇训练笔记包含多个动作段落
- 一个动作段落包含结构化计划行和自由文本备注
- 一个动作可以在历史中聚合来自多篇训练笔记的数据
- 模板本质是可复用的训练笔记草稿
- 训练中的完成进度以独立草稿状态保存，并与训练笔记正文一起恢复

## 6. MVP 功能优先级

### 6.1 P0
必须进入第一版的能力：

- 训练笔记编辑器
- `@动作` 自动补全
- 半结构化解析
- `20 x 8 x 5` 语法识别
- 训练中独立草稿进度状态恢复
- 每条计划行的独立完成按钮
- 累进式圆形进度展示
- 点击后实时改写正文
- 自由文本备注保留
- 动作历史页
- 动作库搜索与自定义
- 全局重量单位设置

### 6.2 P1
建议作为第一版后续增强：

- 文本模板
- 从模板开始训练
- 更丰富的计划行语法
- 更顺滑的动作编辑体验

### 6.3 P2
明确放到后续版本：

- 趋势图
- PR / 1RM
- 训练量分析
- 复杂计划系统
- 更宽松自然语言解析
- Apple Watch / HealthKit

## 7. 核心场景验证

### 7.1 训练中快速记录
用户在健身房打开 App，新建训练笔记，输入：

```text
@卧推
60 x 8 x 4
```

系统会在该行右侧显示一个圆形进度按钮，用户每点击一次表示完成 1 组，圆圈数字和环形进度随之更新，但正文始终保持 `60 x 8 x 4`。如果最终完成 3 组，结束训练后该行保存为 `60 x 8 x 3`。过程中可补充备注。

### 7.2 从模板开始训练
用户提前建好“推胸日”模板。训练当天一键生成当次笔记，在原模板基础上边练边记录完成进度，必要时修改某条计划行内容。

### 7.3 回看动作历史
用户进入“卧推”动作详情页，可以快速看到历次训练日期、计划行完成情况和备注，比翻普通笔记更高效。

### 7.4 管理自定义动作
用户新增自己的动作命名方式，之后在输入 `@` 时可以自动补全，保证记录语言符合自己的习惯。

## 8. 成功标准

如果第一版是成功的，至少应满足：

- 用户愿意在真实训练中持续使用，而不是只在首次尝试时觉得新鲜
- 用户记录训练时不会觉得输入负担太大
- 用户回看某个动作时，效率明显高于翻普通笔记
- 用户既能保留训练中的文字语境，又能获得基本结构化数据

---

## 9. 当前确定的默认决策

- 平台限定为 iPhone
- 面向规律训练者
- 优先力量训练 / 增肌场景
- 产品定位为“笔记优先”
- 使用场景以“边练边记”为主，同时支持轻模板
- 模板采用“文本模板”
- 解析策略采用“半结构化”
- 动作库采用“内置常用 + 用户可编辑”
- 动作详情页优先展示“历史记录列表”
- 重量单位采用“全局默认单位”
- 完成机制采用“按计划行累计独立草稿进度，修改计划行即重置，结束训练后收敛为实际完成记录”
- 高级分析与复杂计划系统不进入 v1

---

## 10. 后续可继续思考的问题
这份方案已经可以作为第一版产品定义基础，接下来建议重点继续思考：

- 训练编辑页里，文本区域和组勾选区域如何组合最顺手
- `20 x 8 x 5` 之外是否还需要支持少量补充语法
- 动作详情页的历史记录展示形式是时间流还是卡片列表
- 模板的创建流程是否足够轻，不会反而让用户觉得麻烦
- 第一版是否需要非常轻量的“今天训练标题 / 标签”能力
