# QuickRep v1 App 架构草案

## 文档目的
这份文档定义 QuickRep 在训练编辑页之外的整体 App 架构，用来约束后续首页导航、历史训练、动作库和偏好设置的实现方向。

当前阶段不立即实施，只先冻结产品骨架，等训练编辑页核心体验开发到一定程度后再接入。

## 总体结构
QuickRep v1 采用 3 个一级导航：

- `训练`
- `动作库`
- `我`

整体原则：

- 训练编辑页仍然是产品核心
- App 其他页面都服务于“开始训练、回看训练、回看动作历史、维护最小偏好”这几件事
- 不在这一阶段引入复杂模板系统、图表分析、账号同步或多端能力

## 1. 训练
`训练` tab 是首页，也是最核心的入口。

### 页面职责
- 展示从新到旧的历史训练时间线
- 提供主按钮：`开始锻炼`
- 点击历史训练条目，进入该次训练的记录详情
- 点击 `开始锻炼`，开始新的锻炼并进入训练编辑页

### 训练列表
- 列表按时间倒序展示训练历史
- 每个条目代表一次训练记录
- 条目至少展示：
  - 训练日期 / 时间
  - 简短摘要，例如动作数或前几个动作名
- 点击条目后，进入该次训练的详情页

### 开始锻炼
- 第一版默认全局只允许 1 条进行中的训练
- 如果已经存在未结束训练，点击主按钮时直接继续当前训练
- 如果不存在未结束训练，则新建一条训练记录并进入训练编辑页

### 训练详情
- 历史训练详情默认只读
- 主要用于回看该次训练的完整记录
- 不在第一版里提供“直接编辑历史训练”的能力

### 与训练编辑页的关系
- 训练编辑页仍然是训练中的主战场
- 从 `训练` tab 点击 `开始锻炼` 或继续当前训练时进入训练编辑页
- 结束训练后，记录从“进行中”转入历史列表
- 历史记录保存的是收敛后的最终文本，而不是训练中间状态

## 2. 动作库
`动作库` tab 用来承载所有可通过 `@` 添加的动作。

### 数据范围
动作库包含两类动作：

- App 内置动作
- 用户自己创建的动作

这两类动作都应出现在训练编辑页的 `@动作` 自动补全中。

### 页面职责
- 展示全部动作
- 支持搜索动作
- 支持新增用户自定义动作
- 支持编辑 / 删除用户自定义动作
- 点击动作后进入动作详情页

### 动作详情页
动作详情页第一版只做轻量信息，不做复杂分析。

包含：

- 该动作的训练历史列表
- 少量摘要数据，例如：
  - 最近一次训练时间
  - 历史训练次数
  - 最高重量
  - 累计完成组数

不包含：

- 趋势图
- PR 曲线
- 1RM 推算
- 复杂统计分析

### 与训练文本的关系
- 动作详情页的数据都从历史训练文本解析得到
- 不额外为动作历史维护一套独立真值
- `rawText` 仍然是训练数据唯一事实源

## 3. 我
`我` tab 用来放用户自己的轻量偏好设置。

### 第一版范围
只包含必要设置，例如：

- 重量单位：`kg` / `lb`

### 设计原则
- 保持极简
- 不引入复杂设置分类
- 不加入与当前核心体验无关的个性化能力

## 数据与状态约束
为了支撑以上结构，训练记录建议具备以下语义状态：

- 进行中的训练
- 已结束的历史训练

建议通过训练记录本身的字段表达，而不是额外引入复杂状态系统。

同时保持以下原则：

- `WorkoutNote.rawText` 仍然是训练内容唯一事实源
- 动作块、计划行、动作历史数据都从文本运行时解析
- 动作库与用户偏好是少量独立持久化数据
- 不单独持久化统计结果

## 路由与页面组织建议
整体 UI 结构建议为：

- 根层使用 `TabView`
- 每个 tab 内部使用独立 `NavigationStack`

建议导航关系：

- `训练`
  - 时间线页
  - 训练编辑页
  - 历史训练详情页
- `动作库`
  - 动作列表页
  - 动作详情页
- `我`
  - 偏好设置页

这样可以保证：

- 一级心智清晰
- 每个 tab 有独立返回栈
- 不把训练编辑流和动作浏览流混在一起

## 第一版明确不做
以下能力不纳入这轮整体架构实施：

- 模板入口
- 图表分析
- PR / 1RM 页面
- 历史训练编辑
- 多条并行进行中的训练
- 账号、同步、云端存储
- Apple Watch / HealthKit 集成

## 实施顺序建议
等训练编辑页做到足够稳定后，再按下面顺序接入：

1. 把 App 根入口改成 3 tab 壳层
2. 给训练记录补齐“进行中 / 已结束”语义
3. 实现 `训练` tab 的时间线和继续当前训练逻辑
4. 实现历史训练只读详情页
5. 实现 `动作库` 列表与最小 CRUD
6. 实现动作详情页的历史列表和轻量摘要
7. 实现 `我` tab 的重量单位偏好

## 默认假设
- 第一版始终只有 1 条进行中的训练
- 历史训练默认只读
- 动作详情页只做轻量摘要，不做图表
- 模板能力后置，不进入当前 App 壳层设计
- 重量单位切换只影响展示与输入上下文，不自动换算旧数据
