# QuickRep 训练文本记录体验开发待办清单

## 1. 目标

这份文档只服务于一个核心能力：

- 把“训练中的文本记录体验”做到极致

它不是完整产品 backlog，也不承担模板、统计、历史分析、复杂动作管理等扩展方向的规划。

当前阶段只有一个判断标准：

- 训练中的文本输入、识别、完成反馈、编辑流畅度，是否已经足够好到可以成为产品的核心竞争力

如果这个问题的答案还不是明确的“是”，就不进入新的核心功能线。

---

## 2. 开发原则

- [ ] 任何任务只有在直接提升输入速度、识别稳定性、进度反馈或编辑流畅度时，才允许进入当前开发阶段
  - 目标：把团队注意力锁死在单一核心能力上
  - 完成标准：本清单中没有模板、统计、历史分析等主线任务
  - 依赖：无
  - 验证方式：逐项检查文档任务，确认均与训练中的文本记录体验直接相关

- [ ] 原始文本作为正文事实源，结构化模型和训练中草稿进度都需围绕文本稳定重建
  - 目标：避免产品心智滑向“表单优先”
  - 完成标准：数据模型和解析器设计都以 `rawText` 为基础，训练中的完成进度以独立草稿状态补充持久化
  - 依赖：P0-2、P0-3、P0-4
  - 验证方式：检查核心类型定义和解析链路，确认结构化数据不独立于原始文本存在

- [ ] 当前阶段不引入复杂纠错、超目标组数、逐组实际差异记录
  - 目标：保持主路径绝对简单
  - 完成标准：P0 范围内只支持计划行累计完成进度
  - 依赖：P0-7、P0-8、P0-9
  - 验证方式：检查交互说明与测试清单，确认不包含撤销、补组、超组、逐组偏差逻辑

## 3. 技术默认决策

- [x] iOS 客户端默认采用 `SwiftUI` 作为应用壳层
  - 目标：快速建立原生应用结构和状态流
  - 完成标准：应用主入口和主要页面容器使用 SwiftUI
  - 依赖：P0-1
  - 验证方式：项目骨架中主 App、Scene、主页面均为 SwiftUI 结构

- [x] 训练编辑器默认采用 `UIKit-backed UITextView` 封装进 SwiftUI
  - 目标：获得行级控制、光标观察、文本范围计算和 overlay 锚定能力
  - 完成标准：编辑器不是纯 `TextEditor`，而是可桥接 UIKit 的自定义组件
  - 依赖：P0-5
  - 验证方式：代码结构中存在 `UIViewRepresentable` 或等价封装，底层承载控件为 `UITextView`

- [ ] 数据持久化默认采用 `SwiftData`
  - 目标：在本地优先的前提下，以较低复杂度完成存储和恢复
  - 完成标准：训练笔记、动作库和独立草稿进度可以本地持久化
  - 依赖：P0-2、P0-12
  - 验证方式：杀进程重开后，训练文本、计划行进度、动作库内容可恢复

- [x] 当前阶段默认单机、本地、无账号、无同步、无网络层
  - 目标：避免技术复杂度掩盖核心体验问题
  - 完成标准：开发任务中不出现登录、接口、云同步相关内容
  - 依赖：无
  - 验证方式：检查项目范围和 backlog，确认没有网络依赖任务

## 4. P0 阻塞项

### P0-1 最小 iOS App 骨架
- [x] 建立最小可运行的 SwiftUI iOS App，单 target、单 scene
  - 目标：得到一个可以持续迭代训练编辑体验的最小宿主
  - 完成标准：打开 App 直接进入训练编辑主页面，无首页导航和多页结构
  - 依赖：无
  - 验证方式：运行 App，首屏即为训练编辑页

### P0-2 核心数据模型
- [x] 定义 `WorkoutNote`
  - 目标：承载训练笔记原始文本和编辑状态
  - 完成标准：包含 `id`、`rawText`、`draftProgressData`、`updatedAt`
  - 依赖：P0-1
  - 验证方式：检查模型定义和持久化映射

- [x] 定义 `ExerciseBlock`
  - 目标：表示一个由 `@动作名` 开始的动作段落
  - 完成标准：包含 `id`、`exerciseName`、`startLineIndex`、`endLineIndex`
  - 依赖：P0-1
  - 验证方式：检查模型定义和解析输出结构

- [x] 定义 `PlanLine`
  - 目标：表示一条被识别出的结构化计划行
  - 完成标准：包含 `id`、`lineIndex`、`exerciseBlockId`、`weight`、`reps`、`targetSets`、`rawText`
  - 依赖：P0-1
  - 验证方式：检查模型定义和解析输出结构

- [x] 定义 `ExerciseLibraryEntry`
  - 目标：支撑 `@动作` 自动补全的数据源
  - 完成标准：包含 `id`、`name`、`isBuiltin`
  - 依赖：P0-1
  - 验证方式：检查模型定义和动作库初始化逻辑

### P0-3 训练笔记文本源模型
- [x] 建立按行拆分的文本源模型
  - 目标：让计划行按钮和解析结构都能稳定绑定到具体行
  - 完成标准：笔记编辑后可得到稳定的行序列和行标识
  - 依赖：P0-2
  - 验证方式：编辑同一篇笔记后，系统仍能正确识别各行类型和关联

- [x] 明确 `rawText` 是正文事实源
  - 目标：把结构化文本状态收敛到正文，把训练中进度收敛到独立草稿状态
  - 完成标准：结构化动作段落、计划行、备注归属都能由 `rawText` 重新推导；训练中进度由独立草稿状态恢复
  - 依赖：P0-2
  - 验证方式：删除缓存结构并重新解析，得到一致结果

### P0-4 半结构化解析器
- [x] 识别 `@动作名` 行
  - 目标：建立动作段落边界
  - 完成标准：遇到合法 `@动作名` 时创建新的 `ExerciseBlock`
  - 依赖：P0-3
  - 验证方式：给定混合文本，输出正确动作分段

- [x] 识别 `重量 x 次数 x 组数` 计划行
  - 目标：提取目标重量、次数和组数
  - 完成标准：例如 `20 x 8 x 5` 被稳定识别为合法 `PlanLine`
  - 依赖：P0-3
  - 验证方式：对合法和非法样例分别运行解析器，检查结果

- [x] 将未命中语法的内容归类为普通备注
  - 目标：保持“笔记优先”的体验
  - 完成标准：自由文本不会被误识别为计划行
  - 依赖：P0-3
  - 验证方式：对备注文本运行解析器，确认只作为普通文本保留

- [x] 输出动作边界、计划行与备注归属
  - 目标：为编辑器 overlay 和后续持久化提供稳定结果
  - 完成标准：解析输出能明确每条计划行隶属的动作段落
  - 依赖：P0-3
  - 验证方式：对多动作、多计划行样例运行解析器，检查归属关系

### P0-5 训练编辑器基础能力
- [x] 使用 `UITextView` 封装训练编辑器组件
  - 目标：获得比 `TextEditor` 更细粒度的文本控制能力
  - 完成标准：编辑器可以桥接到 SwiftUI 页面中使用
  - 依赖：P0-1
  - 验证方式：在训练编辑页中完成真实输入和状态同步

- [x] 支持多行编辑、光标定位和文本范围读取
  - 目标：为行级解析和右侧按钮定位打基础
  - 完成标准：程序可获取当前光标位置、当前行范围和目标行的可视区域
  - 依赖：P0-5
  - 验证方式：在调试和测试中输出当前行信息并与实际光标一致

- [x] 支持检测“用户离开了哪一行”
  - 目标：为延迟校验机制提供触发点
  - 完成标准：从一行移动到另一行时，系统可感知上一行已结束编辑
  - 依赖：P0-5
  - 验证方式：切换光标到其他行时触发行结束回调

### P0-6 `@动作` 自动补全
- [x] 在当前 token 为 `@...` 时触发候选建议
  - 目标：减少动作名输入成本
  - 完成标准：输入 `@卧` 时可以看到相关动作建议
  - 依赖：P0-5、P0-11
  - 验证方式：手动输入触发词，确认候选列表出现

- [x] 从本地动作库提供静态内置和用户自定义动作候选
  - 目标：让补全具备可持续的数据来源
  - 完成标准：候选同时包含内置动作和已创建动作
  - 依赖：P0-11
  - 验证方式：新增动作后再次触发补全，能看到新增项

- [x] 选中补全后替换当前 token，不打断输入流
  - 目标：让补全是提速手段，不是打断手段
  - 完成标准：插入动作名后光标位置合理，用户可继续输入下一行
  - 依赖：P0-5、P0-11
  - 验证方式：选择候选后继续输入计划行，文本结果正确

### P0-7 计划行右侧独立圆形进度按钮
- [x] 仅为合法计划行渲染右侧按钮
  - 目标：保证 UI 反馈只出现在用户可理解的位置
  - 完成标准：动作标题行和备注行不渲染按钮
  - 依赖：P0-4、P0-5
  - 验证方式：混合文本场景下，只有计划行右侧出现按钮

- [x] 为每条计划行渲染独立按钮并锚定到对应行右侧
  - 目标：建立“一行一个进度按钮”的明确心智
  - 完成标准：多条计划行同时存在时，各自按钮位置正确
  - 依赖：P0-4、P0-5
  - 验证方式：滚动与多行场景下，按钮与对应文本行保持一致

- [x] 点击按钮时更新该行独立草稿进度
  - 目标：让每条计划行都具备独立的完成反馈
  - 完成标准：`20 x 8 x 5` 点击一次后圆圈显示 `1/5`，再次点击会继续递增直到上限，正文保持不变
  - 依赖：P0-2、P0-4、P0-5
  - 验证方式：`20 x 8 x 5` 点击 5 次后达到 100%，第 6 次无效

### P0-8 计划行进度与文本绑定规则
- [x] 进度只绑定计划行，不绑定备注
  - 目标：避免自由文本编辑影响训练进度
  - 完成标准：修改计划行下方备注时，该行圆圈进度保持不变
  - 依赖：P0-4、P0-7
  - 验证方式：补充或修改备注后，按钮状态保持一致

- [x] 修改计划行文本内容时立即清空该行进度
  - 目标：确保进度只对应当前文本定义的计划
  - 完成标准：例如把 `20 x 8 x 5` 改为 `20 x 8 x 4` 后，该行草稿进度被清空并按新计划重新记录
  - 依赖：P0-4、P0-7
  - 验证方式：修改计划行后检查正文保持新的纯计划态，按钮状态回到 0

- [x] 结束训练时按实际完成情况收敛计划行
  - 目标：让最终历史记录直接等于实际完成结果
  - 完成标准：`20 x 8 x 5` 且实际完成 4 组时保存为 `20 x 8 x 4`，全部完成时保存为 `20 x 8 x 5`，未完成的 `20 x 8 x 5` 被删除
  - 依赖：P0-4、P0-7、P0-8
  - 验证方式：结束训练后检查最终保存文本是否符合收敛规则

### P0-9 延迟校验机制
- [x] 编辑中的非法中间状态不立即销毁按钮
  - 目标：避免用户编辑计划行时出现视觉闪烁
  - 完成标准：从 `20 x 8 x 5` 改到中途的 `20 x 8 x` 时，按钮暂时保留
  - 依赖：P0-5、P0-7、P0-8
  - 验证方式：逐字符编辑计划行时，按钮不频繁消失重建

- [x] 仅在用户离开当前行后重新判断合法性
  - 目标：让校验时机贴合真实编辑过程
  - 完成标准：离开当前行时触发一次统一解析和状态更新
  - 依赖：P0-5、P0-7、P0-8
  - 验证方式：移动光标到其他行时触发状态变更

- [x] 如果离开后该行非法，则移除按钮并清空进度
  - 目标：维持计划行与按钮的一致性
  - 完成标准：非法计划行不再保留旧草稿进度
  - 依赖：P0-5、P0-7、P0-8
  - 验证方式：将合法行改成非法文本并离开该行，按钮消失且草稿进度被清理

### P0-10 同一动作下多条计划行的独立进度
- [x] 为同一动作段落中的每条计划行分别维护独立草稿进度
  - 目标：支持一个动作下多条计划，而不引入动作级总进度
  - 完成标准：每条计划行的点击只更新自身那一行的草稿进度，不影响其他计划行
  - 依赖：P0-4、P0-7、P0-8
  - 验证方式：在同一动作下录入两条计划行，分别点击并检查状态

### P0-11 最小本地动作库
- [ ] 内置一组少量常用动作
  - 目标：确保自动补全在首次使用时就可用
  - 完成标准：应用首次启动即可补全常见动作
  - 依赖：P0-2
  - 验证方式：首次安装后输入 `@` 能看到候选

- [ ] 支持用户新增、编辑、删除动作
  - 目标：保持长期可用性，但不引入复杂管理能力
  - 完成标准：用户可对本地动作库做基础 CRUD
  - 依赖：P0-2、P0-6
  - 验证方式：新增、修改、删除动作后，补全结果同步更新

- [ ] 支持动作库持久化
  - 目标：让自动补全数据在重启后仍然存在
  - 完成标准：动作库改动在重启 App 后保留
  - 依赖：P0-2、P0-12
  - 验证方式：重启后重新触发补全，确认数据未丢失

### P0-12 最小持久化与恢复
- [x] 持久化训练笔记原始文本
  - 目标：让用户回到 App 时能继续编辑未完成训练
  - 完成标准：最近一篇训练笔记在重开后仍然存在
  - 依赖：P0-2、P0-3
  - 验证方式：杀进程重开后 `rawText` 恢复

- [x] 从 `rawText` 与草稿状态恢复计划行和进行中进度
  - 目标：避免每次打开都丢失训练中的完成状态
  - 完成标准：重启后 `rawText` 与独立草稿进度一同恢复，计划行可继续点击
  - 依赖：P0-2、P0-4、P0-7、P0-8、P0-9
  - 验证方式：点击若干进度并保存草稿状态后重启，计划行状态保持一致

- [ ] 恢复训练文本、计划行进度和动作库
  - 目标：完成第一阶段的最小闭环
  - 完成标准：App 杀进程后重开，文本、进度、动作库均完整恢复
  - 依赖：P0-11、P0-12
  - 验证方式：完整走一遍记录流程后重启检查状态
