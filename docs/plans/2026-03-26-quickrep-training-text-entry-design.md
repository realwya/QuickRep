# QuickRep 训练中文本记录体验专项设计

## 1. 文档目标

这份文档只定义一个核心能力：把“训练中的文本记录体验”做到极致。

它不是完整产品方案，也不讨论长期路线图。它的作用是约束产品前期的注意力，避免在模板、统计、历史页、动作管理等方向上分散精力。

前期产品判断标准只有一个：

- 训练时的记录体验是否已经好到足以成为产品的核心竞争力

如果这个能力还没有做到满意，就不继续扩展新的核心功能。

---

## 2. 核心原则

### 2.1 单点突破
前期只死磕一个能力：

- 用户在真实训练过程中，能不能用接近写笔记的方式，快速、稳定、无打断地记录训练

这比功能数量更重要。

### 2.2 优先体验，不优先功能列表
任何新功能都必须回答一个问题：

- 它是否直接提升训练中的记录体验

如果答案不是明确的“是”，就不应该进入当前阶段。

### 2.3 保持产品心智简单
用户对产品的理解应该非常单一：

- 这是一款能在训练时像写笔记一样记录内容，但又能自动识别训练数据的工具

不是训练计划器，不是数据分析器，也不是传统表格式记录器。

### 2.4 允许克制
前期可以接受“功能不全”，但不能接受“核心体验不顺”。

宁可暂时没有模板、没有图表、没有复杂历史页，也不能接受：

- 输入慢
- 编辑别扭
- 识别不稳定
- 进度反馈混乱
- 训练时需要频繁跳转页面

---

## 3. 核心能力定义

### 3.1 核心任务
用户在训练中完成以下闭环：

1. 输入动作
2. 输入计划行
3. 记录训练过程中的完成进度
4. 补充自由备注
5. 在不打断节奏的前提下继续下一项训练

只要这个闭环还不够顺，就不扩展新的核心能力。

### 3.2 核心体验标准
这个能力做到位，至少应该满足：

- 用户可以很快开始记录，不需要学习复杂操作
- 文本输入仍然像在写笔记，而不是在填表
- 结构化识别足够明确，不会频繁误判
- 训练过程中的完成反馈足够轻量、直观
- 用户在编辑文本时不会因为 UI 抖动、状态丢失而烦躁
- 绝大多数操作都能在一个页面内完成

---

## 4. 交互范围

### 4.1 当前只做的能力
- 一次训练是一篇笔记
- 通过 `@动作名` 输入动作并自动补全
- 通过固定语法输入计划行，例如 `20 x 8 x 5`
- 识别计划行中的重量、次数、组数
- 在计划行右侧显示独立的圆形完成按钮
- 通过点击按钮累计完成进度，并把进度保存在独立草稿状态中
- 保留普通文本作为自由备注
- 结束训练时将计划行收敛为实际完成记录

### 4.2 暂时不作为重点的能力
以下能力可以存在最简版本，但不能成为当前阶段的主要设计重心：

- 模板
- 动作历史页
- 动作库管理
- 设置项

以下能力明确不进入当前专项范围：

- 统计图表
- PR / 1RM
- 周期计划
- 复杂自然语言理解
- 多端同步扩展体验

---

## 5. 核心交互模型

### 5.1 笔记是唯一主界面
训练记录的主战场就是一篇笔记。

用户不应该在训练过程中频繁进入卡片、表单、弹窗或多层页面。所有关键交互都尽量围绕这篇笔记完成。

### 5.2 动作输入
- 用户通过 `@动作名` 开始一个动作段落
- 输入时出现自动补全，减少输入成本
- 从当前动作开始，到下一个动作之前的内容，都归属于当前动作

示例：

```text
@卧推
20 x 8 x 5
最后两组感觉很重

@上斜哑铃卧推
24 x 10 x 4
```

### 5.3 计划行输入
- 采用半结构化语法
- 第一版核心格式为 `重量 x 次数 x 组数`
- 例如 `20 x 8 x 5`
- 只有明确命中规则的行，才会被识别为计划行
- 其余文本一律视为备注，不强行猜测

这个约束非常重要。文本记录体验的前提不是“尽量聪明”，而是“尽量稳定”。

### 5.4 完成进度按钮
- 每条计划行右侧各有一个独立圆形按钮
- 按钮贴近屏幕右边缘，方便训练中重复点击
- 每点击一次表示完成 1 组
- 进度以上限为目标组数的圆形进度条形式反馈
- 每次点击后，系统只更新该行的圆圈数字和环形进度，不改写正文
- 达到目标组数后显示 100%

例如：

```text
20 x 8 x 5
```

首次点击后，圆圈数字会显示为 `1`；继续点击到第 3 次时，圆圈数字会显示为 `3`。这条计划行的按钮最多只能点击 5 次，而正文始终保持 `20 x 8 x 5`。

### 5.5 进度绑定规则
- 进度只绑定当前计划行本身
- 同一动作下多条计划行，各自独立计数
- 计划行下方的备注不影响该行进度
- 修改备注时，进度保持不变
- 训练中的完成进度以独立草稿状态为唯一事实源
- 修改计划行内容本身时，进度直接清空，正文保持为新的纯计划态

例如：

如果用户把某条已有进度的计划改成新的计划，例如：

```text
22.5 x 8 x 5
```

系统应将其视为一条新的纯计划行，而不是继承旧的完成进度。

### 5.6 延迟校验规则
- 用户编辑计划行时，即使中间状态暂时非法，按钮先保留
- 系统不在逐字符输入时频繁销毁或重建按钮
- 只有当用户离开这一行后，才重新判断该行是否仍然合法
- 如果离开该行后不再合法，则移除按钮并清空该行草稿进度

这条规则的目标不是技术正确性，而是避免编辑时的视觉闪烁和状态焦虑。

### 5.7 结束训练时的收敛规则
- 训练结束时，系统会把计划行和独立草稿进度一起收敛为最终实际完成记录
- 如果某行有部分完成，例如目标是 `20 x 8 x 5` 且实际完成 4 组，最终保存为 `20 x 8 x 4`
- 如果某行全部完成，例如目标是 `20 x 8 x 5` 且实际完成 5 组，最终保存为 `20 x 8 x 5`
- 如果某行完全没有完成，例如目标是 `20 x 8 x 5` 且实际完成 0 组，结束训练时直接删除该行
- 最终历史记录只保留实际完成结果，不保留原目标组数痕迹

## 6. 为核心体验服务的设计取舍

### 6.1 为什么不用“每组一个勾选框”
因为那会让界面迅速变得像表单，而不是笔记。

使用单个圆形累计进度按钮的原因：

- 视觉更轻
- 操作路径更短
- 更适合单手连续点击
- 更符合“在笔记上附着反馈”的感觉

### 6.2 为什么不支持复杂纠错
第一阶段不处理：

- 点错撤回
- 少做一组
- 多做一组
- 按组记录实际差异

原因不是这些场景不重要，而是它们会迅速把交互从“极简记录”拉向“复杂状态管理”。

前期应优先确保主路径绝对顺畅：

- 输入计划行
- 点击记录完成
- 在正文里直接看到当前完成情况
- 继续训练

### 6.3 为什么不做宽松自然语言解析
因为训练中的记录体验最怕误判。

比起“看起来聪明”，用户更需要：

- 知道什么会被识别
- 知道什么不会被识别
- 知道为什么按钮会出现或消失

明确规则能建立信任，信任比花哨更重要。

---

## 7. 非目标

这份专项设计当前不追求：

- 帮用户制定训练计划
- 帮用户分析长期进步
- 帮用户管理庞大动作库
- 帮用户生成复杂可视化报表
- 覆盖所有训练类型

如果这些能力开始影响训练中文本记录体验的设计纯度，就应该被推迟。

---

## 8. 验收标准

只有当以下问题基本都能回答“是”时，才说明这个核心能力做到了足够好：

- 用户第一次打开就能理解怎么开始记录吗
- 用户在训练中能主要依赖键盘和少量点击完成记录吗
- `@动作` 和计划行识别是否足够稳定
- 右侧圆形进度按钮是否一眼就能理解
- 进度不写回正文后，用户是否仍能直观理解当前完成情况
- 用户编辑文本时，界面是否足够稳定，不会频繁闪烁或误清空
- 用户是否能在同一篇笔记里自然地混合动作、计划行和备注
- 整个记录过程是否真的比传统表格式录入更轻松

---

## 9. 当前阶段的产品约束

在这个专项没有做到满意之前，后续产品决策应遵守以下约束：

- 不新增新的核心功能线
- 不用模板系统来掩盖记录体验不足
- 不用统计分析来替代核心输入体验
- 不因为“功能看起来太少”而提前扩张
- 所有迭代都优先提升输入速度、识别稳定性、进度反馈和编辑流畅度
- 所有迭代都应维持“训练中以独立草稿状态展示实时完成情况，结束时收敛为最终记录”的语义一致性

---

## 10. 一句话定义

QuickRep 前期最重要的不是“记录更多内容”，而是把下面这件事做到别人做不到：

- 用户在训练时，像写一篇普通笔记一样输入内容，系统能稳定识别动作与计划，把进度独立保存在圆圈状态里，并在结束训练时留下实际完成结果
