# AGENT.md

## Project

Gymlog 是一个基于纯文本的 iOS 健身记录 App。当前阶段只聚焦一个核心能力：

- 把训练中的文本记录体验做到极致

不要把它当成完整健身平台来设计。前期不是做模板系统、统计系统或复杂训练管理，而是把“像记笔记一样记录训练，同时稳定识别动作与计划行，并给出轻量完成反馈”这条主路径打磨到足够好。

## Current Focus

当前唯一主线是训练编辑器里的记录体验，包括：

- `@动作名` 自动补全
- `重量 x 次数 x 组数` 计划行识别
- 计划行右侧独立圆形进度按钮
- 训练中的完成进度以独立草稿状态存在，并显示在右侧圆圈上
- 备注与计划行的分离
- 修改计划行时的进度重置
- 结束训练时按实际完成情况收敛计划行
- 离开当前行后再校验的延迟校验机制

不要主动扩展到以下方向，除非文档被明确更新：

- 模板
- 历史页
- 统计 / 图表
- PR / 1RM
- 周期计划
- 宽松自然语言理解
- 同步、账号、云端存储

## Working Rules

- 原始文本是正文事实源。结构化数据必须可以从文本重建。
- 训练中的完成进度是独立草稿状态，并在结束训练前与 `rawText` 一起持久化。
- 不再把 `n/m` 进度后缀写回正文，也不再兼容把它当作计划行语法。
- 任何新增工作都必须直接提升输入速度、识别稳定性、进度反馈或编辑流畅度。
- 如果一个想法主要是在“补功能”，而不是在“打磨主路径”，默认不做。
- 优先保持规则明确和可预期，不要为了“更聪明”引入高误判率行为。
- 不要实现复杂纠错能力，例如撤销误点、超目标组数、逐组实际差异，除非文档已更新。
- 不要为了局部方便破坏产品心智，不要把编辑器逐步做成表单。
- 最终保存的训练记录应体现“实际完成结果”，而不是原目标组数。

## Core Path Guardrails

- 训练编辑器的击键、选区变化、滚动回调都属于主路径热区；不要在这些路径里同步持久化或叠加不必要的重计算。
- 同一次文本变更里，优先复用同一轮已经得到的 snapshot、解析结果和布局数据，不要重复做可合并的全量工作。
- `View` 负责渲染和交互绑定；文本、解析结果、草稿进度、保存时机这些编辑状态不要长期堆在单个 screen 中。
- 训练编辑主路径里不要用 `try?` 静默吞掉持久化、编码或解码错误；至少在 debug 下输出明确日志。
- 对编辑流畅度有影响的改动，默认补一类交互级验证：计划行修改后的进度行为、结束训练收敛、保存失败恢复，或每击键不会触发持久化。

## Implementation Defaults

当前默认技术方向已经确定，除非文档更新，否则按以下前提工作：

- iOS 原生，`SwiftUI` 作为应用壳层
- 训练编辑器采用 `UIKit-backed UITextView` 封装进 SwiftUI
- 本地优先，单机版本，无账号、无同步、无网络层
- 持久化默认使用 `SwiftData`

更细的实现拆分、优先级和验收条件，直接看：

- [训练文本记录体验开发待办清单](./docs/plans/2026-03-26-gymlog-training-text-entry-backlog.md)

## How To Work

- 做新功能前，先判断它是否属于当前主线。
- 做实现前，先在 backlog 中找到对应任务；如果没有对应任务，先更新文档而不是直接编码。
- 做交互调整时，优先更新专项设计文档，再更新 backlog。
- 做范围判断时，优先删减而不是扩张。
- 做完成声明前，先验证实际结果，不要凭推测说“应该可以”。
